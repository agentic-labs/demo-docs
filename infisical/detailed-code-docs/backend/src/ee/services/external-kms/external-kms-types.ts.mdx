## High-level description
This TypeScript file defines a set of data transfer object (DTO) types for managing external Key Management Service (KMS) operations. These types are used to structure the data for creating, updating, deleting, listing, and retrieving external KMS configurations, incorporating organization permissions and provider-specific details.

## Symbols

### `TCreateExternalKmsDTO`
#### Description
Defines the structure for creating a new external KMS configuration.

#### Inputs
| Name | Type | Description |
|:-----|:-----|:------------|
| slug | string (optional) | A unique identifier for the KMS configuration |
| description | string (optional) | A description of the KMS configuration |
| provider | TExternalKmsInputSchema | Provider-specific input schema |
| (spread) | Omit&lt;TOrgPermission, "orgId"&gt; | Organization permissions excluding orgId |

### `TUpdateExternalKmsDTO`
#### Description
Defines the structure for updating an existing external KMS configuration.

#### Inputs
| Name | Type | Description |
|:-----|:-----|:------------|
| id | string | The unique identifier of the KMS configuration to update |
| slug | string (optional) | A new unique identifier for the KMS configuration |
| description | string (optional) | A new description of the KMS configuration |
| provider | TExternalKmsInputUpdateSchema (optional) | Updated provider-specific input schema |
| (spread) | Omit&lt;TOrgPermission, "orgId"&gt; | Organization permissions excluding orgId |

## Notes
1. All DTO types, except for TListExternalKmsDTO, include specific fields related to the operation (e.g., id, slug, description) along with organization permissions.
2. The organization permissions are included using the spread operator, but the 'orgId' is omitted, suggesting that it's handled separately or not needed in these DTOs.
3. The provider-specific schemas (TExternalKmsInputSchema and TExternalKmsInputUpdateSchema) are imported from a separate file, indicating a modular approach to handling different KMS providers.
4. The use of optional fields (marked with '?') in TCreateExternalKmsDTO and TUpdateExternalKmsDTO allows for flexibility when creating or updating KMS configurations.

This file serves as a central location for defining the data structures used in external KMS operations, ensuring consistency and type safety across the application.
